## 서버 1대에서 시작

처음엔 단순하다.

```
유저 → 서버 1대 → DB 1개
```

서버가 웹 요청도 받고, DB도 들고 있다. 유저가 1명이면 아무 문제 없다.
유저가 늘기 시작하면 이 구조는 하나씩 무너진다. 1장은 그 진화 과정이다.

---

## 캐시 — Memcached API로 파고들기

책에서 캐시 계층이 나오자마자 Memcached API를 직접 물어봤다. 개념보다 실제 명령어를 먼저 보면 이해가 빠르다.

Memcached는 명령어가 단순하다.

| 명령어 | 동작 |
|--------|------|
| `set` | 있으면 **덮어씀** |
| `add` | **없을 때만** 저장 |
| `get` | 조회 |
| `delete` | 삭제 |

여기서 질문이 생겼다. `set`이랑 `add`를 왜 구분해서 쓸까?

**Read-Through에서는 `add`를 써야 한다.**

캐시 미스가 났을 때 두 요청이 동시에 DB로 달려가는 상황을 생각해보면 이해된다.

```
요청 A → 캐시 미스 → DB 조회 중...
요청 B → 캐시 미스 → DB 조회 중...
요청 A → 캐시에 set (저장)
요청 B → 캐시에 set → 요청 A가 저장한 값을 덮어씀 ❌
```

`add`를 쓰면 두 번째 저장은 조용히 실패한다. 이미 있으니까. 불필요한 덮어쓰기를 막는 것이다.

**Write-Through에서는 반드시 `set`을 써야 한다.**

쓰기 전략은 반대다. Write-Through는 DB에 쓸 때 캐시도 동시에 갱신하는 전략이다. 이미 있는 값을 최신으로 업데이트하는 게 목적이니 `set`으로 무조건 덮어써야 한다. `add`를 쓰면 이미 키가 있을 때 저장이 안 되고, 캐시가 구버전으로 남는다.

## 캐시 전략 — Read/Write-Through만 알면 충분할까?

Read-Through와 Write-Through를 공부하다 이런 구분이 생겼다.

> "이 둘의 차이는 캐시 미스에서 언제 캐시를 업데이트하느냐 아닌가?"

정확히는 아니다. 구분 기준은 **읽기냐 쓰기냐**다.

- **Read-Through** — 읽기 요청에서 캐시 미스 발생 시 DB에서 읽어 캐시를 채움
- **Write-Through** — 쓰기 요청 시 캐시와 DB를 동시에 갱신

두 전략은 서로 독립적이라 함께 쓸 수 있다. 그리고 여기서 하나 더 있다.

**Write-Back.** 캐시에만 먼저 쓰고, DB는 나중에 비동기로 저장하는 전략이다.

Write-Back이 유리한 상황이 궁금했다.

> "쓰기가 폭발적으로 많은데 모든 쓰기를 DB에 즉시 반영할 필요가 없을 때"

게임 경험치가 좋은 예다. 몬스터 잡을 때마다 경험치 +10. 1분에 수백 번 업데이트가 발생한다. 매번 DB에 쓰면 DB가 버티질 못한다. 캐시에서 누적하다가 주기적으로 DB에 반영하면 된다. 좋아요 카운터, 실시간 위치 추적도 같은 맥락이다.

반대로 결제, 주문, 재고는 Write-Through다. 유실이 용납되지 않는다.

> 핵심 판단: "이 데이터가 유실되면 얼마나 치명적인가?"

## 캐시 계층 위치 — 브라우저 캐시, CDN, Redis 뭐가 다른가

책에 "캐시를 어떤 계층에 두느냐"는 말이 나온다. 처음엔 뭔 소린지 몰랐다.

```
클라이언트
  ↓
브라우저 캐시 (내 컴퓨터)
  ↓
CDN (네트워크 엣지, 전 세계 공유)
  ↓
인메모리 캐시 Redis/Memcached (서버 제어)
  ↓
DB
```

계층마다 **제어권**이 다르다.

브라우저 캐시는 클라이언트 것이다. 서버가 `Cache-Control` 헤더로 힌트를 줄 수는 있지만, 만료 전엔 서버에 요청 자체를 안 보낸다. 서버가 데이터를 바꿔도 클라이언트는 모른다. 배포 후 구버전 JS/CSS 문제가 여기서 생긴다.

CDN은 정적 콘텐츠를 전 세계 엣지 서버에 캐싱한다. 브라우저 캐시랑 뭐가 다르냐고 물어봤다.

> "CDN이 정적 콘텐츠 캐싱이라고 볼 수 있나? 그럼 브라우저 캐시네?"

결정적인 차이는 **공유 여부**다.

```
브라우저 캐시: 유저 A만 사용 → 유저 B는 여전히 오리진에서 다운로드
CDN:          유저 A가 받으면 엣지에 저장 → 유저 B도 CDN에서 받음
```

CDN은 모든 유저가 공유하는 캐시다.

실제로 한 페이지를 열 때 이미지는 CDN에서, API 데이터는 Redis에서 **동시에** 온다. 각자 역할 분담이다.

## 세션 쿠키는 브라우저 캐시인가?

책에서 NoSQL로 세션을 저장한다는 내용이 나왔다. 그러다 자연스럽게 이 질문이 생겼다.

> "세션 쿠키가 브라우저 캐시로 보면 되는거야?"

아니다. 목적이 다르다.

| | 브라우저 캐시 | 쿠키 |
|--|-------------|------|
| 목적 | **성능** (재요청 줄이기) | **상태 유지** (서버가 클라이언트 식별) |
| 서버 전송 | ❌ 자동 전송 안 함 | ✅ 매 요청마다 자동 전송 |

쿠키는 세션 ID(열쇠)만 가지고 있고, 실제 세션 데이터는 서버의 Redis나 NoSQL에 있다.

---

## 무상태 아키텍처 — 그림 1-14가 말하는 것

책의 그림 1-14를 보며 NoSQL이 두 데이터센터 밖에 독립적으로 있는 게 눈에 들어왔다.

> "상태정보를 따로 저장하는 NoSQL을 따로 둔다는 말이야?"

맞다. 서버가 상태를 가지고 있으면 수평 확장이 안 된다.

```
문제:
유저 → 서버 1 (세션 저장)
유저 재요청 → 서버 2로 가면 → 세션 없음 → 로그인 풀림 ❌
```

세션을 서버 메모리에서 꺼내 **공유 저장소**로 옮기면 어느 서버로 가든 같은 세션을 조회할 수 있다. 덕분에 서버를 마음대로 늘리고 줄일 수 있다 — 그림의 ① 자동 규모 확장이 가능한 이유다.

그림에서 NoSQL이 두 DC 밖에 있는 이유도 여기 있다. DC2가 죽어도 NoSQL은 살아있다. 세션이 유지된다.

## DB 다중화 — 주/부는 데이터가 전부 복사된다

DB 다중화 설명에서 중요한 확인이 있었다.

> "주 부 관계로 나눌 때 일단 모든 전체 데이터가 두 개 다 들어있겠네? 분산 데이터베이스 전략이 아니고?"

정확하다. 다중화와 샤딩은 다르다.

```
다중화 (Replication):
주 DB: [A, B, C, D, E]
부 DB: [A, B, C, D, E]  ← 전체 복사본
→ 목적: 가용성 + 읽기 성능

샤딩 (Sharding):
DB 1: [A, B]
DB 2: [C, D]
DB 3: [E]              ← 데이터를 나눠서 저장
→ 목적: 쓰기 성능 + 용량 확장
```

동기화는 **바이너리 로그(binlog)**로 한다. 주 DB의 모든 변경 쿼리가 로그에 순서대로 기록되고, 부 DB가 그걸 구독해서 동일하게 재실행한다. 비동기 방식이라 약간의 Replication Lag이 생긴다.

방금 내가 바꾼 데이터를 부 DB에서 조회하면 아직 구버전일 수 있다. 이걸 Read-Your-Writes 문제라 한다. 중요한 읽기는 주 DB에서 하는 게 안전하다.

## 메시지 큐 — 소비자가 뭔데?

메시지 큐 설명에서 "소비자"라는 단어가 처음 나왔을 때 물어봤다.

> "소비자가 뭐야?"

큐에서 메시지를 꺼내 처리하는 서버 또는 프로세스다.

```
생산자 (앞단, 유저 요청 받음) → [메시지 큐] → 소비자 (뒤단, 실제 처리)
```

유튜브라면 영상 업로드를 받는 웹 서버가 생산자, 1080p/720p/480p로 인코딩하는 서버가 소비자다.

> "이런 식이면 서버가 2단계로 나뉘는 거지? 앞단에 소비자가 있고 뒤에 생산자가 있는건가?"

반대다. 생산자가 앞단이다. 그리고 소비자는 뒤단에서 독립적으로 돌아간다.

핵심은 **유저에게 빠르게 응답**하고 오래 걸리는 작업은 뒤에서 조용히 처리하는 구조다. ack/nack이 메시지 유실을 막아준다. TCP의 ack와 같은 철학이다 — 계층만 다를 뿐.

## 1장 마무리 역질문 — 셋 중 하나만 쓴다면?

챕터 끝에 역질문을 받았다.

> "캐시, 메시지 큐, NoSQL 중 하나만 쓸 수 있다면 어떤 걸 선택하겠어?"

캐시를 골랐다. 이유는 하나였다.

> "레이턴시에 가장 큰 영향을 주는 DB 읽기 작업을 줄이는 게 전체적인 성능 향상이 가장 클 것 같아"

맞다. 메시지 큐는 특정 작업만 빠르게, NoSQL은 특정 데이터만 빠르게. 캐시는 전체 시스템을 빠르게 만든다.

```
캐시 히트율 80% 가정:
DB 요청 10,000건 → 캐시가 8,000건 차단 → DB는 2,000건만 처리
DB 100ms → 캐시 1ms → 전체 응답시간 극적으로 개선
```

## 1장 핵심 원칙 두 가지

모든 진화 과정을 관통하는 원칙이 두 가지다.

**다중화 — 가용성**
모든 계층에 SPOF가 없어야 한다. 로드밸런서, 웹 서버, 캐시, DB, 데이터센터 전부.

**독립적 분리 — 확장성 + 장애 격리**
각 컴포넌트가 독립적으로 스케일 아웃되고, 하나가 죽어도 나머지는 살아있어야 한다.

이 두 가지가 합쳐져야 수백만 사용자를 감당하는 시스템이 된다.
