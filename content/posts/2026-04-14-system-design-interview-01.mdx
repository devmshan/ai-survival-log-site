## 서버 1대에서 시작

처음엔 단순하다.

```
유저 → 서버 1대 → DB 1개
```

서버가 웹 요청도 받고, DB도 들고 있다. 유저가 1명이면 아무 문제 없다.

유저가 늘기 시작하면 이 구조는 하나씩 무너진다. 1장은 그 진화 과정을 따라간다.

---

## 웹 계층과 데이터 계층 분리

첫 번째 변화. 서버에서 DB를 떼어낸다.

```
유저 → 웹 서버 (요청 처리)
              → DB 서버 (데이터 저장)
```

왜? 웹 서버와 DB는 **리소스를 쓰는 방식이 다르다.** 웹 서버는 CPU, DB는 디스크 I/O. 한 서버에 두면 서로 방해한다. 분리하면 각자 최적화할 수 있다.

---

## 수직 확장 vs 수평 확장

유저가 더 늘었다. 서버가 버티질 못한다.

**수직 확장 (Scale Up)** — 서버 스펙을 올린다.
```
서버 1대 (4코어 16GB) → 서버 1대 (32코어 256GB)
```
단순하다. 코드 변경 없이 그냥 빵빵한 서버로 교체하면 된다.
하지만 한계가 있다. 아무리 비싼 서버도 끝이 있다. 그리고 1대니까 죽으면 전체가 죽는다.

**수평 확장 (Scale Out)** — 서버를 여러 대 늘린다.
```
서버 1대 → 서버 N대 (로드밸런서로 분산)
```
이론적으로 무한 확장 가능. 대규모 시스템의 표준 방향이다.

---

## 로드밸런서

수평 확장을 하려면 앞단에 **로드밸런서**가 필요하다.

```
유저 → 로드밸런서 → 서버 1
                  → 서버 2
                  → 서버 3
```

요청을 여러 서버에 골고루 분배한다. 서버 1이 죽으면? 로드밸런서가 감지하고 서버 2, 3으로만 보낸다. 유저는 모른다.

로드밸런서 자체도 죽으면 안 되니까 Active-Standby로 2대 운영한다. **SPOF(단일 장애점) 제거**가 핵심 원칙이다.

---

## DB 다중화

웹 서버는 여러 대가 됐다. 근데 DB는 아직 1대다. 병목이 여기로 옮겨간다.

**주/부(Master/Replica) 구조:**
```
쓰기 → 주 DB
읽기 → 부 DB 1, 부 DB 2, 부 DB N
```

읽기가 쓰기보다 훨씬 많다. 읽기를 여러 부 DB에 분산하면 부하가 확 줄어든다.

동기화는 **바이너리 로그(binlog)**로 한다. 주 DB의 모든 변경 쿼리가 로그에 기록되고, 부 DB가 그걸 구독해서 동일하게 재실행한다.

주의할 점 하나. 비동기 복제라 **Replication Lag**이 생긴다. 방금 내가 바꾼 데이터를 부 DB에서 조회하면 아직 구버전일 수 있다. 중요한 읽기는 주 DB에서 하는 게 안전하다 (Read-Your-Writes 패턴).

## 캐시

DB 앞에 캐시 계층을 둔다.

```
웹 서버 → 캐시(Redis) → 캐시 히트면 즉시 반환
                      → 캐시 미스면 DB 조회 후 캐시 저장
```

DB 응답 100ms, 캐시 응답 1ms. 히트율 80%면 DB 요청이 20%로 줄어든다.

캐시 전략이 여러 가지다.

- **Read-Through:** 읽기 미스 시 DB에서 가져와 캐시에 저장
- **Write-Through:** 쓸 때 캐시 + DB 동시 업데이트. 항상 최신값 보장. DB 쓰기만큼 느리다는 단점
- **Write-Back:** 캐시에만 먼저 쓰고 DB는 나중에 비동기 저장. 빠르지만 캐시 장애 시 유실 위험

캐시 일관성의 핵심 질문:

> "이 데이터가 몇 초 오래돼도 괜찮은가?"

괜찮으면 TTL로 단순하게, 절대 안 되면 명시적 무효화로.

## CDN

정적 콘텐츠(이미지, JS, CSS)는 CDN으로 뺀다.

```
유저 → CDN (가까운 엣지 서버에서 응답)
     → 원본 서버 (CDN에 없을 때만)
```

캐시는 동적 데이터, 서버가 제어. CDN은 정적 콘텐츠, 모든 유저가 공유.

한 페이지를 열 때 이미지는 CDN에서, API 데이터는 캐시에서 동시에 온다.

---

## 무상태 아키텍처

수평 확장을 제대로 하려면 서버가 **상태를 갖고 있으면 안 된다.**

```
문제:
유저 → 서버 1 (세션 저장)
유저 재요청 → 서버 2 → 세션 없음 → 로그인 풀림 ❌
```

해결책: 세션을 서버 메모리에서 꺼내 **공유 저장소(Redis/NoSQL)**로 옮긴다.

```
서버 1 \
서버 2  → Redis (세션 공유 저장소)
서버 3 /
```

어느 서버로 가든 같은 세션을 조회한다. 덕분에 서버를 자유롭게 늘리고 줄일 수 있다.

## 메시지 큐

오래 걸리는 작업(이미지 처리, 이메일 발송, 문서 생성)을 메인 흐름에서 분리한다.

```
생산자 (앞단) → [메시지 큐] → 소비자 (뒤단)
```

생산자는 큐에 넣고 즉시 응답한다. 소비자가 알아서 꺼내 처리한다. 유저는 기다리지 않아도 된다.

**ack/nack**이 메시지 유실을 막는다. 처리 완료 후 ack를 보내야 큐에서 삭제된다. ack 없이 소비자가 죽으면 큐가 다른 소비자에게 재전달한다.

---

## 샤딩

DB 데이터 자체가 너무 많아지면 수평 샤딩을 고려한다.

```
user_id 1~1000만      → 샤드 1
user_id 1000만~2000만 → 샤드 2
user_id 2000만~       → 샤드 3
```

보통 수직 샤딩(도메인 분리)으로 시작해서 데이터가 더 늘면 수평 샤딩으로 간다. 샤딩 후에는 샤드 간 조인이 어렵다는 트레이드오프가 있다.

---

## 1장 핵심 원칙 두 가지

모든 진화 과정을 관통하는 원칙이 두 가지다.

**다중화 — 가용성**
모든 계층에 SPOF가 없어야 한다. 로드밸런서, 웹 서버, 캐시, DB, 데이터센터 전부.

**독립적 분리 — 확장성 + 장애 격리**
각 컴포넌트가 독립적으로 스케일 아웃되고, 하나가 죽어도 나머지는 살아있어야 한다.

이 두 가지가 합쳐져야 수백만 사용자를 감당하는 시스템이 된다.
