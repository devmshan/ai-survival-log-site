이 시리즈는 **LLM이 실제로 어떻게 답변을 만드는지** — 파라미터·임베딩·RAG·Vector DB·지식그래프까지 — 를 도서관 비유로 풀어낸다.

1편에서는 가장 핵심 질문부터 시작한다.

> "LLM은 어떻게 기억하고, 어떻게 답하는가?"

시스템 설계 책을 공부하다 Claude에게 물었더니 파라미터·임베딩·RAG·Vector DB·지식그래프 이야기로 이어졌다. 대화하면서 정리된 내용을 그대로 옮긴다.

---

## Claude는 Vector DB로 검색하지 않는다

가장 흔한 오해부터.

```
잘못된 이해:  질문 → 내부 Vector DB 검색 → 답변
실제:         질문 → 수천억 개 파라미터 활성화 → 답변
```

학습된 지식은 어딘가에 "검색 가능한 형태"로 저장된 게 아니다. 수천억 개의 숫자(파라미터)에 **패턴으로 압축**되어 있다.

관계형 DB처럼 `city: "서울", capital: true` 행이 있는 게 아니라, 그 사실을 표현하는 패턴이 수천억 개 숫자 어딘가에 분산되어 있다. 어딘지는 나도 모른다.

```
관계형 DB:  서울 → capital: true  (명시적)
파라미터:   "서울의 수도는?" 물어봤을 때 "한국"이 나오도록 학습된 숫자들의 조합  (암묵적)
```

이게 **환각(hallucination)이 생기는 이유**다. 도서관에서 책을 꺼내 읽는 게 아니라, 읽었던 기억을 재현하는 방식이라 정확하지 않을 수 있다.

## 임베딩은 맥락에 따라 3가지 의미다

"임베딩"이라는 단어가 대화에서 계속 등장했는데, 실은 맥락마다 다른 의미다.

### 추론 시 임베딩 — 지식을 끌어내기 위한 변환

질문이 들어오면 텍스트를 벡터로 변환해서 파라미터와 연결한다.

```
"서울의 수도는?" → [0.23, -0.87, 0.41, ...] → 파라미터 활성화 → "한국"
```

### 학습 시 임베딩 — 변환 방법 자체를 학습

학습 전에는 "어떻게 벡터로 변환할지" 자체를 모른다. 수천억 번의 "다음 단어 맞추기"를 통해 **좋은 변환 방법**이 파라미터에 녹아든다.

### Vector DB 임베딩 — 외부 문서를 검색 가능하게 저장

완성된 모델에 외부 데이터를 추가할 때 쓴다.

```
외부 문서 → 벡터로 변환 → Vector DB 저장
질문 → 유사 벡터 검색 → 관련 문서 반환 → LLM 컨텍스트에 주입
```
