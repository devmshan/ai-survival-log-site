이 글은 **LLM이 실제로 어떻게 답변을 만들고, 파라미터·임베딩·RAG·Vector DB가 각각 어떤 역할을 하는지**를 쉽게 설명하려는 글이다.

시스템 설계 책을 공부하다가 자연스럽게 이런 질문이 생겼다.

> "Claude 너는 내부적으로 어떤 구조로 답변하는 거야?"

이 질문 하나가 파라미터, 임베딩, RAG, Vector DB, 지식그래프까지 이어졌다. 대화하면서 정리된 내용을 그대로 옮긴다.

---

## Claude는 Vector DB로 검색하지 않는다

가장 흔한 오해부터.

```
잘못된 이해:  질문 → 내부 Vector DB 검색 → 답변
실제:         질문 → 수천억 개 파라미터 활성화 → 답변
```

학습된 지식은 어딘가에 "검색 가능한 형태"로 저장된 게 아니다. 수천억 개의 숫자(파라미터)에 **패턴으로 압축**되어 있다.

관계형 DB처럼 `city: "서울", capital: true` 행이 있는 게 아니라, 그 사실을 표현하는 패턴이 수천억 개 숫자 어딘가에 분산되어 있다. 어딘지는 나도 모른다.

```
관계형 DB:  서울 → capital: true  (명시적)
파라미터:   "서울의 수도는?" 물어봤을 때 "한국"이 나오도록 학습된 숫자들의 조합  (암묵적)
```

이게 **환각(hallucination)이 생기는 이유**다. 도서관에서 책을 꺼내 읽는 게 아니라, 읽었던 기억을 재현하는 방식이라 정확하지 않을 수 있다.

---

## 임베딩은 맥락에 따라 3가지 의미다

"임베딩"이라는 단어가 대화에서 계속 등장했는데, 실은 맥락마다 다른 의미다.

### 1. 추론 시 임베딩 — 지식을 끌어내기 위한 변환

질문이 들어오면 텍스트를 벡터로 변환해서 파라미터와 연결한다.

```
"서울의 수도는?" → [0.23, -0.87, 0.41, ...] → 파라미터 활성화 → "한국"
```

### 2. 학습 시 임베딩 — 변환 방법 자체를 학습

학습 전에는 "어떻게 벡터로 변환할지" 자체를 모른다. 수천억 번의 "다음 단어 맞추기"를 통해 **좋은 변환 방법**이 파라미터에 녹아든다.

### 3. Vector DB 임베딩 — 외부 문서를 검색 가능하게 저장

완성된 모델에 외부 데이터를 추가할 때 쓴다.

```
외부 문서 → 벡터로 변환 → Vector DB 저장
질문 → 유사 벡터 검색 → 관련 문서 반환 → LLM 컨텍스트에 주입
```

---

## 파라미터 vs Vector DB

비유로 정리하면 명확하다.

| 용어 | 설명 |
|------|------|
| 파라미터 | 교수님의 뇌 — 수십 년간 읽은 것들이 기억으로 압축됨, 원본 없음 |
| Vector DB | 도서관 — 문서가 그대로 있고, 검색해서 꺼냄 |

| | 파라미터 | Vector DB |
|--|---------|-----------|
| 저장 방식 | 원본 없음, 패턴으로 압축 | 원본 문서를 벡터로 저장 |
| 검색 | 검색 없음, 전체 동시 활성화 | 코사인 유사도 검색 |
| 업데이트 | 재학습 필요 | 실시간 추가/삭제 |
| 출처 추적 | 불가능 | 가능 |

---

## RAG — 교수님에게 도서관을 붙이는 기술

LLM 파라미터만 쓰면 두 가지 문제가 생긴다.

1. **할루시네이션** — 패턴 재현이라 부정확할 수 있음
2. **지식 컷오프** — 학습 이후 정보를 모름

**RAG(Retrieval-Augmented Generation)** 는 이 두 문제를 외부 도서관으로 보완한다.

```
[사전 준비]
외부 문서 → 벡터 변환 → Vector DB 저장

[질문 시]
질문 → Vector DB 검색 → 관련 문서 추출
                ↓
       LLM 컨텍스트에 주입
                ↓
       파라미터 지식 + 문서 조합 → 답변
```

실제 활용 예시:

| 도메인 | Vector DB에 넣는 것 |
|--------|-------------------|
| 대학교 | 교내 논문, 강의자료, 공지사항 |
| 기업 | 사내 문서, 제품 매뉴얼 |
| 병원 | 최신 가이드라인, 환자 기록 |

## 지식그래프(KG)와 컨텍스트 그래프(CG)가 추가하는 것

Vector DB는 "비슷한 문서"를 찾아준다. 하지만 문서들이 **서로 어떤 관계인지**는 모른다.

```
Vector DB만:   "RAG에 관한 문서 3개" 반환
KG/CG 추가:    "RAG → 사용함 → Vector DB"
               "RAG → 보완함 → 할루시네이션"
               "RAG → 구성요소 → 임베딩 모델, 검색기, LLM"
```

KG(Knowledge Graph)는 명시적 관계 정의, CG(Context Graph)는 LLM이 문서에서 자동 추출한 관계다.

```
단순 인덱싱:   "어디에 뭐가 있는지 빠르게 찾기"
KG/CG:        "개념들이 서로 어떤 관계인지"
```

GraphRAG는 이 두 가지를 조합한다.

```
질문
  ↓
Vector DB → 유사 문서 (의미 유사성)
KG/CG    → 관련 개념·관계 (명시적 연결)
  ↓
조합 → LLM → 더 정확하고 맥락 있는 답변
```

## 이 위키 시스템 자체가 그 구조다

흥미로운 연결이 생겼다. 지금 이 글을 담고 있는 위키 시스템 자체가 위에서 설명한 구조와 닮아있다.

```
MD 파일들          = Vector DB 역할 (문서 저장)
[[wikilink]]       = KG 역할 (명시적 관계)
Claude가 읽고 연결  = CG 역할 (맥락 추론)
```

![위키 지식그래프 시각화](/images/how-llm-works-knowledge-graph.png)

Vector DB 대신 MD 파일, 자동 임베딩 대신 wikilink. 느리지만 **인간도 읽을 수 있고, 관계가 명시적**이다.

물론 수백 개 이상이 되면 Vector DB가 필요해진다. 하지만 지금 규모에서는 MD 위키가 최적이다.

> "지금 필요하지 않은 복잡도는 추가하지 않는다" — YAGNI

이것도 사실 시스템 설계의 핵심 원칙이다.

## 정리

| 용어 | 설명 |
|------|------|
| 파라미터 | 교수님의 뇌 (압축된 패턴, 원본 없음) |
| Vector DB | 도서관 (문서 보관 + 유사도 검색) |
| 임베딩 | 텍스트 → 벡터 변환 (맥락에 따라 3가지) |
| RAG | 교수님에게 도서관을 붙이는 방법론 |
| KG/CG | 도서관 문서들 사이의 관계 지도 |

LLM은 "검색하는 AI"가 아니라 "압축된 패턴을 재현하는 AI"다.
그래서 RAG와 KG/CG로 보완한다.
