# Content SEO Guide

## 목적

이 문서는 `content/posts`에 새 글을 쓰거나 기존 글을 수정할 때 따라야 하는 콘텐츠 SEO 작성 규칙을 정의한다.

대상:

- 수동 작성 글
- upstream `ai-survival-log` publish 산출물 검토/후처리
- 기존 글 리라이트 작업

이 가이드는 downstream site repo 기준 문서이지만, 실제 글 작성과 시리즈 기획이 upstream `ai-survival-log`에서 먼저 일어나는 경우가 많다.
따라서 글 작성 규칙, 시리즈 구성 규칙, 검색형 제목/요약 원칙은 upstream repo에도 같은 수준으로 반영되어야 한다.

이 문서는 기술 SEO가 아니라 글 자체의 검색 대응력에 관한 규칙을 다룬다.

## Scope Boundary

이 가이드는 블로그형 텍스트 콘텐츠와 그 변형에 우선 적용한다.

자동으로 포함되지 않는 대상:

- YouTube 스크립트
- 웹툰 스토리보드
- 발표자료 슬라이드
- 비블로그형 소셜 자산

이런 lane는 원칙을 재사용할 수는 있지만, 공식 workflow가 되면 별도 lane 규칙이나 contract를 가져야 한다.

## 기본 원칙

- 검색 유입은 우연히 생기지 않는다. 글의 주제, 제목, 요약, 도입부가 검색 의도와 직접 맞물려야 한다.
- 브랜드 톤은 유지할 수 있지만, 검색형 글은 감성형 제목만 단독으로 두지 않는다.
- 독자가 첫 10초 안에 “이 글이 무엇을 설명하는가”를 이해할 수 있어야 한다.
- 한 글이 끝에서 끊기지 않게 관련 글과 시리즈로 연결해야 한다.
- 모든 글이 같은 방식으로 검색을 노릴 필요는 없지만, 검색 유입을 기대하는 글은 아래 규칙을 기본으로 따른다.

## Enforcement Levels

이 문서는 선택 가이드가 아니라 운영 규칙이다.

### `warn`

- 제목, 요약, 도입부가 더 검색 친화적으로 개선될 여지가 있음
- 관련 글 연결이 얕지만 완전히 비어 있지는 않음
- `thumbnail`이 없어도 렌더링은 되지만 공유 품질은 약해짐

### `block`

- 발행 글인데 주제가 제목/요약/도입부 어디에서도 분명히 드러나지 않음
- tone-first 제목인데 필요한 `seoTitle`이 없음
- `description` 또는 `seoDescription`이 실질 주제를 설명하지 못함
- 내부 링크가 날짜 접두사 없는 잘못된 경로를 사용함
- `## 함께 읽으면 좋은 글` 섹션이 없음

### `escalate`

- upstream `ai-survival-log`와 다른 SEO 작성 규칙을 의도적으로 채택함
- `seoTitle`, `seoDescription`, canonical handling의 의미를 바꿈
- 관련 글/시리즈 탐색 규칙을 콘텐츠 계약 수준에서 바꿈

## 글 유형 분류

새 글이나 기존 글을 수정할 때 먼저 유형을 정한다.

### 1. 검색형 글

예:

- 개념 설명
- 비교 글
- 가이드
- 문제 해결
- 도구 정리

규칙:

- 검색 키워드를 제목 또는 `seoTitle`에 직접 포함
- 도입부 첫 문단에서 글의 질문과 답변 범위를 명확히 설명
- `description` 또는 `seoDescription`에 요약을 명시

### 2. 혼합형 글

예:

- 개인 경험 + 기술 설명
- 회고 + 실무 팁

규칙:

- `title`은 톤을 살릴 수 있음
- 대신 `seoTitle`, `seoDescription`은 검색형 문장으로 보완
- 도입부에서 개인 서사보다 글의 실질 주제를 먼저 드러내는 편이 좋다

### 3. 브랜딩/회고형 글

예:

- 시작글
- 철학/회고
- 다짐/에세이

규칙:

- 검색 유입보다 브랜드 톤이 우선일 수 있음
- 그래도 `description`은 읽는 사람이 내용을 예측할 수 있게 써야 함
- 관련 기술 글로 이어지는 링크를 넣어 사이트 전체 탐색성을 높인다

## 제목 규칙

### `title`

- 화면에 보이는 기본 제목
- 블로그 톤과 개성을 유지할 수 있다
- 단, 완전히 추상적이거나 의미를 짐작할 수 없는 제목은 피한다

### `seoTitle`

- 검색 노출을 위해 필요할 때만 추가하는 선택 필드
- 아래 상황이면 도입을 우선 검토한다:
  - 기본 제목이 너무 감성적임
  - 핵심 개념어가 제목에 없음
  - 사용자가 검색할 표현과 제목이 너무 멀음

좋은 예:

- `LLM은 실제로 어떻게 작동할까? 파라미터, 임베딩, RAG 쉽게 이해하기`
- `Claude Code에서 Codex로 갈아탄 이유와 차이점 정리`
- `ECC 완전 가이드: 설치, 사용법, 주의할 점까지`

다음 조건이면 `seoTitle` 누락을 `block`으로 본다:

- 기본 제목이 감성적이거나 서사 중심이다
- 검색자가 찾을 핵심 개념이 제목 앞부분에 없다
- 도입부만 읽어도 개념이 늦게 드러난다

피할 예:

- 개념어가 전혀 없는 추상 제목
- 질문만 있고 답의 범위가 없는 제목
- 너무 넓은 제목

## 설명문 규칙

### `description`

- 기본 요약 문장
- 가능한 한 1~2문장, 짧고 명확하게 쓴다
- “이 글이 무엇을 설명하는지”를 드러내야 한다

### `seoDescription`

- 검색결과 요약을 더 명확히 제어하고 싶을 때 사용
- 다음 상황에서 우선 검토:
  - 기본 설명이 감성적이거나 모호함
  - 검색 키워드를 더 직접 반영해야 함
  - CTR 개선 여지가 있음

좋은 기준:

- 핵심 주제 포함
- 독자가 얻는 내용 포함
- 과장 없이 구체적

기본 `description`이 톤 유지에는 성공했지만 검색 의도 설명에는 실패하면 `seoDescription`을 추가하거나 기본 설명 자체를 고친다.

## 섹션 헤더 규칙

`##` 헤더는 그 섹션에서 독자가 얻는 **인사이트/발견/전환점**을 전달해야 한다. 단순한 라벨이면 안 된다.

**번호 접두사 금지:** `## 1. 설치`, `## 2. 설정` 형식은 모든 글 유형에서 사용하지 않는다. 가이드/레퍼런스형 글도 예외 없음.

나쁜 예 (라벨):
- `## 3. 핵심 개념` — 번호 + 라벨
- `## 설치하고 돌려봤다` — 행동만 기술

좋은 예 (인사이트):
- `## 핵심 개념` — 번호 없이 명확한 제목
- `## 설치 한 줄로 되는데 Rules는 수동이다` — 발견을 전달

테스트: 헤더만 읽어서 글의 논지가 따라오는가?

## 도입부 규칙

첫 2~3문단은 아래 역할을 해야 한다.

- 이 글이 다루는 질문
- 어디까지 설명하는지
- 누구에게 유용한지

권장 구조:

1. 문제/질문 제시
2. 이 글이 답할 범위 제시
3. 본문으로 이어지는 짧은 맥락

피할 패턴:

- 서사만 길고 주제가 늦게 나오는 시작
- 검색형 글인데 결론이 3~4문단 뒤에 나오는 구조

## 내부 링크 규칙

새 글 또는 수정 글에는 가능한 한 아래를 포함한다.

- 관련 글 2개 이상
- 시리즈가 있으면 시리즈 허브 또는 앞/뒤 글 연결
- 토픽 클러스터 안의 진입 글 또는 심화 글 연결

연결 방식:

- 본문 안 자연스러운 문장 링크
- 글 하단 관련 글
- 시리즈 패널/탐색

규칙:

- 같은 태그만 달고 끝내지 않는다
- 독자가 다음 글로 이동할 이유가 보여야 한다

### 내부 링크 slug 형식

내부 포스트 링크는 반드시 날짜 접두사가 포함된 전체 slug를 사용한다:

```
✓ /posts/2026-04-15-claude-code-to-codex
✗ /posts/claude-code-to-codex
```

날짜 접두사 없는 slug는 라우팅 키와 일치하지 않아 404가 발생한다.

### "함께 읽으면 좋은 글" 섹션

모든 발행 글은 `## 함께 읽으면 좋은 글` 섹션을 포함해야 하며 최소 2~3개 링크를 수동으로 작성한다.

이 섹션은 아래의 자동 생성 "관련 글" 카드 섹션과 별개다. 태그가 다른 글과 겹치지 않으면 자동 섹션이 노출되지 않으므로, 수동 섹션이 유일한 독자 탐색 경로가 된다.

### 관련 글 자동 생성 로직

사이트는 태그/시리즈 기반으로 "관련 글" 카드를 자동 생성한다:

| 신호 | 점수 |
|---|---|
| 같은 시리즈 (`seriesSlug` 일치) | +100 |
| 공통 태그 1개당 | +10 |
| 점수 = 0 | 섹션 미표시 |

태그가 다른 글과 고립된 글은 "관련 글" 섹션이 아예 표시되지 않는다. 이런 경우 "함께 읽으면 좋은 글" 수동 섹션이 반드시 필요하다.

발행 글에서 이 섹션이 없으면 `block`이다. 자동 관련 글 카드가 있으니 생략해도 된다고 판단하지 않는다.

## 토픽 클러스터 규칙

우선 토픽:

- LLM 기초
- 임베딩 / Vector DB / RAG
- 시스템 설계
- AI coding workflow / automation

운영 방식:

- 각 토픽마다 진입 글 1개를 만든다
- 진입 글은 개요/지도 역할을 한다
- 세부 글은 진입 글과 서로 링크한다
- 시리즈로 묶을 수 있으면 시리즈 메타를 사용한다

## 대표 이미지 규칙

- `thumbnail`은 카드 이미지이자 현재 social preview 대표 이미지로 사용된다
- 가능하면 글 주제를 설명하는 이미지나 도식 이미지를 넣는다
- 이미지가 없으면 기본 fallback이 사용되지만, 검색/공유 품질은 낮아질 수 있다

엣지 케이스:

- `thumbnail`이 있지만 실제 파일이 없음
- 기본 fallback만 있고 글 공유 품질을 기대하는 경우
- 이미지 경로가 계약과 다르게 작성됨

## 작성 체크리스트

새 글을 발행하거나 기존 글을 크게 수정할 때 아래를 확인한다.

- [ ] 글 유형을 정했는가
- [ ] 제목이 주제를 충분히 드러내는가
- [ ] 필요하면 `seoTitle`을 분리했는가
- [ ] `description`이 요약 역할을 하는가
- [ ] 필요하면 `seoDescription`을 분리했는가
- [ ] 첫 2~3문단 안에 글의 핵심 주제가 드러나는가
- [ ] `thumbnail`을 지정했는가
- [ ] 관련 글 2개 이상 연결했는가
- [ ] 모든 내부 링크가 날짜 접두사 포함 전체 slug 형식인가 (`/posts/YYYY-MM-DD-slug`)
- [ ] `## 함께 읽으면 좋은 글` 섹션이 있는가 (최소 2개 링크)
- [ ] `##` 헤더에 번호 접두사(`## 1.`, `## 2.`)가 없는가
- [ ] 시리즈화 가능성이 있으면 `series`, `seriesSlug`, `seriesOrder`를 검토했는가

## 엣지 케이스

- tone-first 제목이 본문 도입부에서도 보완되지 않음
- `description`이 감상은 전달하지만 주제를 설명하지 않음
- 태그가 고립되어 자동 관련 글이 비어 있는데 수동 링크도 없음
- 시리즈 글인데 현재 글의 위치가 설명되지 않음
- downstream 후처리 과정에서 upstream 제목/요약 규칙과 drift가 생김

## 리라이트 우선순위 기준

기존 글을 고칠 때는 아래 순서로 우선순위를 준다.

1. 검색 의도가 뚜렷한 기술 글
2. 시리즈/클러스터 중심 글
3. 검색형으로 전환 가능한 혼합형 글
4. 브랜딩/회고형 글

판단 기준:

- 검색 키워드가 명확한가
- 이미 내부 링크 허브 역할을 할 수 있는가
- 제목/설명 수정 시 CTR 개선 여지가 큰가
